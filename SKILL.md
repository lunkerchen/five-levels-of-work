# Five Levels of Work

源自 Nikhyl Singhal 的高主動性框架。核心命題：**L4 是門檻，L5 是目標。**

## 五層定義

| 等級 | 行為 | 一句話 |
|------|------|--------|
| L1 | 「有問題。」 | 純通報 |
| L2 | 「有問題，我找到一些原因。」 | 通報+診斷 |
| L3 | 「問題是 X，原因可能是 A/B，解法有 C/D。」 | 診斷+方案選項 |
| L4 | 「問題是 X，我認為主因是 A，建議解法 C。」 | 診斷+推薦 |
| L5 | 「我發現問題→診斷→修復→順便跟你說一聲。」 | 自主解決+知會 |

## 應用原則

### 對 Hermes 的指令格式

你對我的指令，最低 L3：

```
[目標] camera-market 登入流程
[問題] OTP 驗證後沒跳轉
[你懷疑] Redis cooldown key 沒過期
[你要的方向] 先確認 Redis state，再決定加 fallback 還是改 TTL
```

L1/L2 的指令我會直接退回要求補 context。

我對你的回報永遠 L5 — 你只會收到「修好了/做好了，摘要在此」。

### 五域對照表

#### 攝影工作室
| 等級 | 實際行為 |
|------|---------|
| L1 | 「這週詢價變少了」 |
| L3 | 「詢價變少，IG reach 掉 40%，可能因為週四沒發文」 |
| L4 | 報價流程自動化：詢價→分類→產生報價→送出，你只確認數字 |
| L5 | 報價全自動，客戶下單後自動排程進行事曆 |

#### C2C 二手器材 (camera-market)
| 等級 | 實際行為 |
|------|---------|
| L1 | 「有個 bug」 |
| L3 | Issue 帶：預期行為/實際行為/日誌/你懷疑的原因 |
| L4 | Issue template 強制帶診斷+推薦修法 |
| L5 | loop-fix 自動偵測→分支修復→發 PR 給你 review |

#### 內容創作
| 等級 | 實際行為 |
|------|---------|
| L1 | 「今天要發文」 |
| L3 | 「這週素材夠拆 3 篇 Threads + 1 支 Shorts」 |
| L4 | 素材進→自動拆成多平台內容→排程到位→你只決定發不發 |
| L5 | 內容日曆自動運轉，你週花 15 分 review |

#### 交易/財務
| 等級 | 實際行為 |
|------|---------|
| L1 | 「看一下大盤」 |
| L3 | 「大盤破月線，外資連三賣，要不要減半導體？」 |
| L4 | 每日報告含持倉分析+建議動作+風險評估 |
| L5 | 策略參數你設，觸發條件自動執行，你只收通知 |

#### 個人生產力
| 等級 | 實際行為 |
|------|---------|
| L1 | 「該整理 obsidian 了」 |
| L3 | 「筆記累積 200 篇未整理，建議 wikijanitor 歸檔」 |
| L4 | 每晚 cron 自動整理+摘要，你早上看 diff |
| L5 | 系統自治，你只處理例外 |

### 品質檢查

審查任何產出或任務時，檢查它落在哪一級：

- **L1-L2** → 退回，要求補診斷和方案
- **L3** → 可接受，但你必須做選擇
- **L4** → 好，你可以直接決定
- **L5** → 理想狀態，只需知會

### L4 模板格式

問題描述時強制使用：

```
☐ 問題已定義（預期 vs 實際）
☐ 原因已診斷（至少一個 root cause）
☐ 方案已列舉（至少 2 個選項）
☐ 推薦已給出（你選哪個 + 為什麼）
```
